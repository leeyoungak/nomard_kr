GitHub 이슈 #$ARGUMENTS 를 분석하고 해결 계획을 수립하세요.

## Step 1 — 이슈 fetch

아래 명령으로 이슈 전체 내용을 가져오세요.

```
gh issue view $ARGUMENTS --json number,title,body,labels,comments,assignees,milestone
```

## Step 2 — 코드베이스 탐색

이슈 내용을 바탕으로 관련 파일과 코드를 탐색하세요.

- 이슈에서 언급된 기능, 컴포넌트, 파일명을 키워드로 검색
- 수정이 필요할 것으로 예상되는 파일을 직접 읽어 현재 구현 파악
- 관련 타입 정의, 인터페이스, 의존 모듈 확인
- 유사한 기존 구현 패턴이 있다면 참고

## Step 3 — 해결 계획 수립

탐색 결과를 바탕으로 아래 형식으로 계획을 작성하세요.

---

### 이슈 요약
이슈 #$ARGUMENTS 의 핵심 문제 또는 요청을 2-3문장으로 요약하세요.

### 원인 분석 (버그인 경우)
문제가 발생하는 근본 원인과 해당 코드 위치를 설명하세요.

### 변경 파일 목록

| 파일 | 작업 | 이유 |
|------|------|------|
| `경로/파일명` | 신규 / 수정 / 삭제 | 변경이 필요한 이유 |

### 단계별 실행 계획

- [ ] **1단계: 단계명**
  - 무엇을: 구체적인 작업 내용
  - 어떻게: 구현 방식 (기존 패턴 참고 등)
  - 검증: `npx tsc --noEmit` / 브라우저 확인 / 테스트 실행 등

(이하 단계 반복)

### 의존 관계

```
1단계
  ↓
2단계 → 3단계 (병렬 가능)
  ↓
4단계
```

### 예상 리스크
- 이 변경이 영향을 줄 수 있는 다른 기능이나 파일
- 주의해야 할 엣지 케이스

## Step 4 — 사용자 검토 요청

계획 작성이 끝나면 **구현을 시작하기 전에** 반드시 사용자에게 다음을 물어보세요.

> 위 계획대로 구현을 진행할까요?
> 수정하고 싶은 부분이 있으면 말씀해주세요.

사용자가 승인하면 단계별로 구현을 시작하세요.
사용자가 수정을 요청하면 계획을 업데이트한 뒤 다시 검토를 요청하세요.
